---
title: Entity Commands
description: Right-click context menu — provider, actions, and dispatch
---

When the player right-clicks an entity in the scene, the engine opens a context menu populated with the actions valid for that entity (Examine, Talk, Attack, Loot, Pick Up…). This is the primary way the player issues commands to the world.

The system spans two engine layers:

::alert{type="info"}
**Across the layers** — the menu *provider* (open/close state + action dispatch) lives in the runtime layer (`@artificer-forge/engine/runtime`), while the *action definitions* (which actions exist per entity type, their labels, icons, and conditions) live in the UI layer (`@artificer-forge/engine/ui`).
::

## The Provider — `useContextMenuProvider`

`useContextMenuProvider` is installed once near the root of your game (the `<Game>` component does this for you). It owns the reactive menu state and a set of action handlers, and `provide()`s an API down the tree:

```ts
import { useContextMenuProvider } from '@artificer-forge/engine/runtime'

const { state, open, close, emitAction } = useContextMenuProvider()
```

| Member | Description |
|--------|-------------|
| `state` | `{ open, entityId, x, y }` — drives the menu's visibility and screen position |
| `open(entityId, x, y)` | Open the menu for an entity at screen coordinates |
| `close()` | Close the menu |
| `emitAction(action, entityId)` | Dispatch a chosen action to all registered handlers |

## The Consumer — `useContextMenu`

Any component rendered under the provider can read or drive the menu via `useContextMenu`. The in-scene `Character` and `Interactable` components call it on right-click:

```ts
import { useContextMenu } from '@artificer-forge/engine/runtime'

const { open } = useContextMenu()

function handleContextMenu(event: TresPointerEvent) {
  event.nativeEvent.preventDefault()
  open(props.entityId, event.nativeEvent.clientX, event.nativeEvent.clientY)
}
```

To react to a chosen action, register a handler with `onAction` — it auto-cleans on unmount:

```ts
const { onAction } = useContextMenu()

onAction((action, entityId) => {
  if (action === 'talk') startDialog(entityId)
  if (action === 'attack') beginCombat(entityId)
})
```

::alert{type="warning"}
`useContextMenu` throws if called outside a `<Game>` (i.e. without a provider above it in the tree). Always mount your scene inside the engine's `Game` component.
::

## The Actions — `useEntityActions`

`useEntityActions` (UI layer) decides *which* actions an entity offers. Each `EntityAction` has an `id`, `label`, `icon`, optional `color`, and an optional `condition(entity)` predicate. Actions are grouped by entity type:

```ts
import { useEntityActions } from '@artificer-forge/engine/ui'

const { getActionsForEntity } = useEntityActions()

// Returns DropdownMenuItem[][] filtered by each action's condition
const items = getActionsForEntity(gameStore.getEntity(entityId))
```

Example of the conditional logic baked into the definitions:

```ts
const characterActions: EntityAction[] = [
  { id: 'examine', label: 'Examine', icon: 'i-heroicons-magnifying-glass' },
  {
    id: 'talk',
    label: 'Talk',
    icon: 'i-heroicons-chat-bubble-left-right',
    condition: entity => !isDeadCharacter(entity) && !entity.hostile,
  },
  {
    id: 'attack',
    label: 'Attack',
    icon: 'i-heroicons-fire',
    color: 'error',
    condition: entity => !isDeadCharacter(entity) && entity.faction !== 'player',
  },
  {
    id: 'loot',
    label: 'Loot',
    icon: 'i-heroicons-hand-raised',
    condition: entity => isDeadCharacter(entity), // dead → lootable
  },
]
```

A dead character drops its combat actions and gains `Loot`; a locked chest offers `Lockpick` instead of `Open`. The condition functions run against live `EntityState`, so the menu always reflects current world state.

## Flow

1. Player right-clicks an entity → the in-scene component calls `open(entityId, x, y)` (runtime provider).
2. The HUD's context menu reads `state` and renders the items from `getActionsForEntity(entity)` (UI actions).
3. Player picks an item → the HUD calls `emitAction(action, entityId)`.
4. Every handler registered via `onAction` runs — your game logic resolves the command (talk, attack, loot…).

::alert{type="info"}
The action *definitions* are data, not behavior. They describe what a player *can* do; wiring an action `id` to actual game logic is the job of your `onAction` handlers.
::
