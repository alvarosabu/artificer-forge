---
title: Armor
description: Physical and magical armor pools, equip-derived maxima, and damage absorption
---

Armor is a two-channel buffer that sits between incoming damage and an entity's HP. Each character carries a **physical** pool and a **magical** pool — depleted by matching damage types, refilled only when gear changes.

## Model

Armor lives on two layers of `EntityState`:

| Field | On | Meaning |
|-------|-------------|---------|
| `armor: { physical?, magical? }` | items | Static contribution from a piece of gear — copied from the YAML template at spawn |
| `physicalArmor`, `magicalArmor` | characters | Current depleting pools. Max is **derived**, never stored |

There is no `maxPhysicalArmor` field. The cap is recomputed from equipped items whenever it's needed, which keeps the source of truth on the gear.

## Core rule: `deriveMaxArmor`

The maths is a pure reducer in `@artificer-forge/engine/core` — no Vue, no Three, no store. It sums the `armor` contribution across a list of equipped items:

```ts
import { deriveMaxArmor } from '@artificer-forge/engine/core'

interface ArmorItem {
  armor?: { physical?: number, magical?: number }
}

// deriveMaxArmor(equipped: ArmorItem[]): { physical: number, magical: number }
const { physical, magical } = deriveMaxArmor(equippedItems)
```

The runtime store wraps this with the entity lookup — see `derivedMaxArmor(entityId)` below.

## Item Schema

Both `physical` and `magical` are optional — a piece can grant one, the other, or both.

```yaml
# content/entities/items/armor/breastplate.yaml
templateId: breastplate
type: item
subtype: armor
name: Steel Breastplate
armor:
  physical: 20
  magical: 3
weight: 20
value: 120
```

```yaml
# content/entities/items/armor/ring-of-warding.yaml
templateId: ring-of-warding
type: item
subtype: ring
name: Ring of Warding
armor:
  physical: 2
  magical: 15
weight: 0.1
value: 200
```

::note
Any equippable item can contribute armor — not just the `armor` slot. A ring, helmet, or amulet that grants `magical: 15` adds to the pool just like a breastplate.
::

## Store API

Exposed on `useGameStore` from `@artificer-forge/engine/runtime`:

```ts
import { useGameStore } from '@artificer-forge/engine/runtime'

// Sum of armor across every equipped item on this character.
// Internally filters equipped items, then calls core deriveMaxArmor.
derivedMaxArmor(entityId: string): { physical: number, magical: number }

// Refill both pools to their derived max. Called on (re)equip — no passive regen.
refreshArmor(entityId: string): void

// Deplete the matching armor pool first; overflow hits HP.
applyDamage(
  targetId: string,
  amount: number,
  damageTypeId: string,
): { armorAbsorbed: number, hpDamage: number }
```

### derivedMaxArmor

Iterates entities, summing `armor.physical` / `armor.magical` for every item where `containerId === entityId` **and** `slot` is set (equipped items only — bag contents don't count).

### refreshArmor

Called automatically from `moveItem` whenever a slotted item changes on either side of the move. Sets `physicalArmor` and `magicalArmor` back to the new max — equipping a heavier breastplate fully refills the physical pool, unequipping it can leave the current pool above the new max only until the next call.

::warning
There is no passive regeneration. Pools only refill via `refreshArmor`, which fires on equip/unequip. If you want out-of-combat regen, add it explicitly.
::

### applyDamage

The single entry point for taking damage. It looks up the damage type's `armorType`, delegates the maths to the pure core `computeDamage(entity, amount, armorType)`, applies the resulting state with `updateEntity`, and returns the split:

```ts
const armorType = useDamageTypeStore().get(damageTypeId)?.armorType
const { armorAbsorbed, hpDamage, next } = computeDamage(entity, amount, armorType)
updateEntity(targetId, next)
// 'physical' → deplete physicalArmor first, overflow → hp
// 'magical'  → deplete magicalArmor first, overflow → hp
// undefined  → goes straight to hp (true damage)
```

Returns `{ armorAbsorbed, hpDamage }` so callers can drive damage numbers and hit reactions differently for absorbed vs. through-armor hits.

## Damage Type Mapping

Each entry in the `damageType` collection declares which armor channel it bites:

```ts
interface DamageTypeDefinition {
  damageTypeId: string
  armorType: 'physical' | 'magical'
  // …
}
```

A type with no `armorType` bypasses armor entirely (true damage). Look up via `useDamageTypeStore().get(damageTypeId)`.

## Lifecycle

```
spawn character with starting equipment
       │
       ▼
spawnItemEntity(..., { slot })   for each equipped piece
       │
       ▼
refreshArmor(characterId)        ← pools filled to derived max
       │
       ▼
combat: applyDamage(target, amt, damageTypeId)
       │
       ├─ armorType matches → drain physicalArmor / magicalArmor
       │
       └─ overflow → hp
       │
       ▼
player equips a new piece via moveItem(...)
       │
       ▼
refreshArmor() runs on the affected character(s)   ← pools refill
```

## Example

```ts
const gameStore = useGameStore()

// Inspect what the character would have at full gear:
const { physical, magical } = gameStore.derivedMaxArmor(playerId)

// A fire spell lands. fire's armorType is 'magical':
const { armorAbsorbed, hpDamage } = gameStore.applyDamage(playerId, 18, 'fire')
// → magicalArmor drained first; remainder removed from hp
```

## UI Surface

### Nameplate

`Nameplate.vue` accepts both pools and their maxima. When `maxPhysicalArmor` or `maxMagicalArmor` is `> 0` it renders, above the HP bar:

- a stat row with `ph:shield-fill` (white, physical) and `ph:shield-star-fill` (blue, magical) reading `current / max`,
- a split progress bar — left half white for physical, right half secondary-color for magical.

The whole block is gated by a `hasArmor` computed and hidden when both maxima are 0, so unarmored creatures show just the HP bar. The bar is also hidden when `isDead`.

```vue
<Nameplate
  :name="entity.name"
  :hp="entity.hp"
  :max-hp="entity.maxHp"
  :physical-armor="entity.physicalArmor"
  :max-physical-armor="maxArmor.physical"
  :magical-armor="entity.magicalArmor"
  :max-magical-armor="maxArmor.magical"
  :status-effects="entity.statusEffects"
/>
```

### Wiring from Actor / Character

`Actor.vue` and `Character.vue` compute `maxArmor` from the store and pass both current pools and maxima to the nameplate. The current values live on `EntityState`; the max is recomputed each render via `derivedMaxArmor` — no caching, no `maxPhysicalArmor` field.

```ts
const maxArmor = computed(() => gameStore.derivedMaxArmor(props.entityId))
```

### Party HUD

The portrait in [`CharacterPanel`](/actors/party-hud) renders the same split armor bar under the HP bar, also gated by `hasArmor`. It reads `member.physicalArmor` / `member.magicalArmor` and `derivedMaxArmor(member.id)` exactly like the nameplate.

### Inventory Modal

`CharacterInventoryModal` shows three stat pills along the top of the preview pane — HP, physical armor, magical armor — using the same icons as the nameplate. Armor pills only render when the corresponding `maxArmor.*` is `> 0`. See [Inventory &rarr; UI Components](/inventory/ui-components).
