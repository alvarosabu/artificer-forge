The combat system coordinates UI slots, store state, 3D pointer events, and character animation into a single attack flow. This page explains the pieces, how they connect, and what happens at each step.

## Combat Flow

A single melee attack involves five distinct phases:

1. **Slot activation** — player clicks the action bar or presses a hotkey
2. **Targeting mode** — cursor becomes a crosshair; hovering enemies highlights them with a red outline
3. **Target confirmation** — clicking a hostile entity triggers the attack sequence
4. **Movement and attack** — the attacker moves to melee range, plays the attack animation, then damage is rolled
5. **Resolution** — damage number pops up, the target plays a hit animation, both characters return to idle

```
Player presses 1
       │
       ▼
useActionBar.activateSlot(0)
       │
       ▼  onSlotActivated hook fires with slot { id: 'attack-melee' }
       │
       ▼
Game.vue → combatStore.enterTargeting()
       │
       ▼  cursor = crosshair, isTargeting = true
       │
       ▼
Character.vue pointer events
  pointerenter → combatStore.hoverTarget(entityId)  → red outline
  pointerleave → combatStore.hoverTarget(null)       → outline removed
  click        → combatStore.confirmTarget(entityId)
       │
       ▼
combatStore.executeAttack(targetId)   [internal]
       │
       ├─ attackerRef.moveTo(meleePos)
       │
       ├─ onArrive: attackerRef.meleeAttack()
       │
       ├─ +800ms: rollDamage() → gameStore.updateEntity(hp)
       │           targetRef.showDamage()
       │           targetRef.play(HIT_A)
       │
       └─ +600ms: both refs play IDLE_A, isAttacking = false
```

## State Machine

The combat store drives a simple three-state machine:

| State | `isTargeting` | `isAttacking` | Description |
|-------|:---:|:---:|-------------|
| **idle** | `false` | `false` | Default, no combat in progress |
| **targeting** | `true` | `false` | Waiting for the player to pick a target |
| **attacking** | `false` | `true` | Attack sequence in progress |

Transitions:

- `idle → targeting`: `enterTargeting()` called from `Game.vue`
- `targeting → attacking`: `confirmTarget()` validates host team, calls `executeAttack()`
- `targeting → idle`: `cancelTargeting()` (Escape key or right-click)
- `attacking → idle`: after hit animation resolves (600ms post-attack)

::alert{type="warning"}
`enterTargeting()` is a no-op when `isAttacking` is `true`. This prevents the player from queuing another attack mid-sequence.
::

## Key Pieces

### `useActionBar`

Shared composable (singleton via `createSharedComposable`) that owns the action slot definitions and keyboard shortcuts 1–0. Fires an event hook when a slot is activated.

```ts
const { onSlotActivated } = useActionBar()

onSlotActivated((slot) => {
  // slot.id === 'attack-melee'
})
```

See [Action Bar](/combat/action-bar) for the full API.

### `Game.vue`

The scene wrapper that bridges the 2D UI and the 3D canvas. It listens to `onSlotActivated` and delegates to the combat store. It also owns the invisible floor mesh that tracks cursor world position during targeting.

```vue
<TresMesh
  @pointermove="(e) => combatStore.updateCursorPoint(e.point)"
  @click="handleFloorClick"
>
```

Right-clicking cancels targeting via a `contextmenu` event listener:

```ts
useEventListener('contextmenu', (e) => {
  if (combatStore.isTargeting) {
    e.preventDefault()
    combatStore.cancelTargeting()
  }
})
```

### `useCombatStore`

Pinia store that owns all targeting state and the `executeAttack` sequence. It communicates with the 3D scene exclusively through `useSceneRefs`.

See [Combat Store](/combat/combat-store) for the full API.

### `Character.vue`

Dumb model renderer that becomes interactive during targeting. When `combatStore.isTargeting` is `true`, pointer events delegate to the store instead of handling selection normally.

```ts
function handlePointerEnter() {
  if (combatStore.isTargeting) {
    combatStore.hoverTarget(props.entityId)
    if (entity?.team === 'hostile') addToSelection(rig.value, 'hostile')
    return
  }
  // normal hover handling ...
}
```

`Character.vue` also `defineExpose`s the methods the combat store calls through `useSceneRefs`:

| Exposed method | Called by |
|---------------|-----------|
| `moveTo(pos)` | `executeAttack` — moves attacker to melee position |
| `onArrive(cb)` | `executeAttack` — fires callback when movement ends |
| `meleeAttack()` | `executeAttack` — plays attack animation |
| `play(anim, blend)` | `executeAttack` — plays hit/idle animations |
| `showDamage(value, type, crit)` | `executeAttack` — spawns damage number HTML overlay |
| `lookAt(point)` | `updateCursorPoint` — rotates leader toward cursor |

### `useSceneRefs`

Bridge between Pinia store logic and live Three.js component instances. Stores registered `CharacterRef` objects by entity ID so the combat store can call methods on them without importing Vue component internals.

```ts
const { getCharacterRef } = useSceneRefs()
const attackerRef = getCharacterRef(attackerId)
attackerRef?.moveTo(meleePos)
```

## Visual Feedback

Three layers of visual feedback reinforce the combat state:

| Feedback | Component | Condition |
|----------|-----------|-----------|
| Crosshair cursor | CSS class on `document.body` | `isTargeting === true` |
| Red outline on enemy | `useOutlinePass` (`hostile` preset) | Pointer over hostile during targeting |
| Target reticle (floor ring) | `<TargetReticle>` in `Game.vue` | `isTargeting && hoveredTargetId !== null` |
| Damage number popup | `<DamageNumber>` in `Character.vue` | After `executeAttack` resolves |

```vue
<!-- Game.vue — reticle follows the hovered target -->
<TargetReticle
  v-if="combatStore.isTargeting && combatStore.hoveredTargetId"
  :position="reticlePosition"
  color="#ff4444"
  :radius="0.8"
/>
```

## Extending Combat

To add a new ability (e.g. a ranged attack or an AoE):

1. Add a new `ActionSlot` entry to `DEFAULT_SLOTS` in `useActionBar.ts` with a unique `id`
2. Listen for it in `Game.vue` inside `onSlotActivated`
3. Add the corresponding logic to `useCombatStore` (new exported function or extend `executeAttack`)
4. If the ability needs a different targeting shape, expose new visual feedback components from `Game.vue`

::alert{type="info"}
The separation between `useActionBar` (UI slots) and `useCombatStore` (game logic) is intentional. Action slots should never import the combat store directly — keep that wiring in `Game.vue`.
::
