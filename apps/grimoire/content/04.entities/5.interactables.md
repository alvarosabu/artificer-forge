Interactables are world objects players can interact with: chests, doors, levers. Unlike characters with baked GLB animations, interactables use **code-driven transforms** - the component programmatically rotates/moves mesh parts based on state.

The renderer is the engine's `Interactable` component, driven by a single `:entity-id` prop:

```vue
<script setup lang="ts">
import { Interactable, useGameStore } from '@artificer-forge/engine/runtime'

const gameStore = useGameStore()
const interactableEntities = computed(() =>
  [...gameStore.entities.values()].filter(e => e.type === 'interactable'),
)
</script>

<template>
  <Suspense>
    <Interactable
      v-for="entity in interactableEntities"
      :key="entity.id"
      :entity-id="entity.id"
    />
  </Suspense>
</template>
```

It reads its model, position, and `animations` config from the store, and right-clicking it opens the [entity context menu](/core-concepts/commands) (Open, Lockpick, Attack, …).

## Template Structure

An interactable template defines what the object is and how it animates:

```yaml
# content/entities/interactables/chest-common.yaml
templateId: chest_common
type: interactable
subtype: container
name: Wooden Chest

model: /models/Chest.glb
animations:
  default: closed
  states:
    closed:
      target: chest_lid
      rotation: { x: 0, y: 0, z: 0 }
    open:
      target: chest_lid
      rotation: { x: -0.6, y: 0, z: 0 }
```

The `animations` block tells the component which mesh to animate (`target`) and what transforms to apply for each state. Rotation values are **PI multipliers** - `-0.6` means `-0.6 * Math.PI` ≈ -108°. This keeps YAML readable without raw radian values.

## Code-Driven Animation Approach

Why not bake animations in the GLB? Two reasons:

1. **Flexibility** - Adjust timing, easing, and angles without re-exporting models
2. **Simplicity** - No rigging required, just named meshes

The component finds the target mesh using `useGraph` from TresJS, which builds a dictionary of named objects from the scene. Instead of traversing the tree looking for "chest_lid", you access it directly via `graph.value?.nodes?.['chest_lid']`.

## Lerped Transforms

State changes don't snap - they interpolate smoothly using `MathUtils.lerp` in the render loop:

```ts
mesh.rotation.x = MathUtils.lerp(
  mesh.rotation.x,      // current value
  targetRotation.x,     // target value
  delta * LERP_SPEED,   // how fast to interpolate
)
```

Each frame moves the rotation closer to the target. Higher `LERP_SPEED` = faster animation. The delta ensures consistent speed regardless of frame rate.

## State Management

The component tracks current state locally and syncs to the store:

1. Click triggers `toggle()` which cycles through states (closed → open → closed)
2. Local `currentState` ref updates, changing `targetRotation`
3. Render loop lerps toward new target
4. Store gets updated with `opened: true/false` for game logic

This pattern works for any number of states - a lever could have off → low → high → off.

## Model Requirements

For code-driven animations:

- **Named meshes** - The animated part needs a name in Blender matching the YAML `target`
- **Export closed** - Model should be in default state, rotations relative to that
- **Apply transforms** - In Blender, apply all transforms before export to avoid offset issues

::warning
Mesh names are case-sensitive. `Lid` ≠ `lid` ≠ `chest_lid`. Check your GLB in a viewer if animations aren't working.
::

## Placing in Scenes

Interactables are spawned via [scenes](/entities/scenes), not hardcoded:

```yaml
# content/scenes/dungeon-entrance.yaml
entities:
  - templateId: chest_common
    position: { x: 2, y: 0, z: 2 }
    rotation: { x: 0, y: 0.25, z: 0 }
    overrides:
      lootTable:
        - id: healing_potion
          chance: 1.0
```

The `overrides` field lets you customize specific instances - this chest always has a potion.

## Containers & Loot

Interactables of `subtype: container` (chests, urns, sarcophagi) act as **containers** in the inventory system. When `spawnFromTemplate` instantiates them, it walks the `lootTable`, rolls each entry, and spawns the surviving items as item entities reparented to the container:

```yaml
lootTable:
  - id: healing_potion       # item templateId
    quantity: [1, 3]         # uniform random range — or a fixed number
    chance: 0.8              # 0.0 – 1.0
```

The resulting items are real entities — `gameStore.itemsIn(chestId)` returns them, the loot popover renders them, and they move via the same `moveItem` primitive as any other item.

### Dead Characters as Containers

A character with `hp <= 0` (or `dead: true`) becomes a lootable container at runtime. No special template work — their existing equipped gear and bag items become accessible via the same loot popover, including equipped slots (see [Loot](/inventory/loot#corpses-as-containers)).

The full chest/corpse/world-pickup flow is documented in the [Loot](/inventory/loot) section of the inventory chapter.
