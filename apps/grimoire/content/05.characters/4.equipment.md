Equipment is the visual layer on top of the [item-entity model](/inventory/overview). When a character has an item entity with `slot === 'mainHand'` in their bag, `useEquipment` finds the item, loads its GLB, and attaches it to the rig's hand bone. Change the slot in the store → the weapon swaps on screen.

## Basic Usage

```vue
<script setup lang="ts">
import { useGLTF } from '@tresjs/cientos'
import { useEquipment, useGameStore } from '@artificer-forge/engine/runtime'

const gameStore = useGameStore()
const { nodes } = useGLTF('/models/characters/hero.glb')
const rig = computed(() => nodes.value?.Rig_Medium)

// Derive { mainHand, offHand } templateIds from equipped item entities.
const equipment = computed(() => gameStore.derivedEquipment(entity.id))
useEquipment(rig, equipment)
</script>
```

Two lines in `Character`. The composable handles the rest.

::note
**Across the layers** — `useEquipment` and the game store live in `@artificer-forge/engine/runtime` (Three/Vue movement and attachment logic). The inventory editing UI — the doll, slots, and cells that drive `equipItem`/`unequipItem` — lives in `@artificer-forge/engine/ui`.
::

## How It Works

1. `useGraph` builds a flat dictionary of named nodes from the rig.
2. Finds bones by slot name (`handslotr` for main hand, `handslotl` for off hand).
3. Loads the weapon GLB via `GLTFLoader` + `DRACOLoader`.
4. Clones the scene and attaches it to the bone with `bone.add(scene)`.
5. Watches the equipment ref reactively — swaps or removes weapons as needed.

## The Composable

```ts
export function useEquipment(
  rig: Ref<TresObject3D | undefined>,
  equipment: Ref<Equipment | undefined>,
)
```

| Param | Type | Description |
|-------|------|-------------|
| `rig` | `Ref<TresObject3D>` | The character's armature/rig |
| `equipment` | `Ref<Equipment>` | `{ mainHand, offHand }` templateIds |

### Equipment Interface

```ts
export interface Equipment {
  mainHand?: string  // weapon templateId
  offHand?: string   // weapon templateId
}
```

::note
This shape is the **rendering input** to `useEquipment` — it isn't stored on the character. The source of truth is the item entities themselves (`type: 'item'`, `containerId: characterId`, `slot: 'mainHand'`). Use `gameStore.derivedEquipment(id)` to project them into the shape `useEquipment` expects.
::

## Bone Mapping

Equipment slots map to bone names in the rig:

| Slot | Bone Name | Description |
|------|-----------|-------------|
| `mainHand` | `handslotr` | Right hand slot |
| `offHand` | `handslotl` | Left hand slot |

::note
Blender exports remove dots from bone names. If your bones are `handslot.r` in Blender, they become `handslotr` in the exported GLB.
::

Other slots (`helmet`, `armor`, `gauntlets`, …) live in the inventory model but **don't currently bone-attach** — they're tracked in the store and appear in the inventory UI, but no mesh swap happens in-world yet.

## Store API

Equip and unequip go through the inventory primitives:

```ts
const gameStore = useGameStore()

// Equip an item that already belongs to this character.
gameStore.equipItem(itemEntityId, 'mainHand')

// Unequip — moves the item back into the same character's bag.
gameStore.unequipItem(itemEntityId)
```

These wrap `moveItem`, which handles validation, equip-swap, capacity, and encumbrance sync. See [Store API](/inventory/store-api) for the full contract.

::warning
The old `equipWeapon(charId, templateId, slot)` and `unequipWeapon(charId, slot)` APIs are gone. Equipping now operates on item **entity ids**, not template ids — because the entity carries quantity, condition, and stack identity that the template alone can't.
::

## Weapon Templates

Weapons are item templates with a `model` field pointing to the GLB:

```yaml
# content/entities/items/weapons/shortsword.yaml
templateId: shortsword
type: item
subtype: weapon
name: Shortsword
icon: /img/items/shortsword.png
model: /models/items/weapons/Sword_1Handed.glb
damage:
  dice: "1d8"
  type: slashing
properties:
  - versatile
weight: 3
value: 15
```

The `model` path is what `useEquipment` loads and attaches to the bone. The `icon` is what the inventory UI shows in cells and tooltips.

## Starting Equipment

Characters declare starting gear in their template. `spawnFromTemplate` reads it and spawns each entry as an item entity reparented to the new character:

```yaml
# content/entities/characters/companions/hero.yaml
templateId: hero
type: character
name: Hero
model: /models/characters/hero.glb
equipmentSlots:
  - mainHand
  - offHand
  - helmet
  - armor
equipment:
  mainHand: shortsword
```

Two fields work together:

- `equipmentSlots` — which slots this character actually has (drives the inventory doll layout)
- `equipment` — which item template fills each slot at spawn

After spawn, the items exist as real entities (`type: 'item'`, `containerId: heroId`, `slot: 'mainHand'`) and can be moved, dropped, or transferred like any other.

## Resource Disposal

The composable handles full cleanup:

- **On unequip** — removes mesh from bone, disposes geometry, materials, and all texture maps.
- **On scope dispose** — cleans up both slots automatically.
- **Race conditions** — per-slot request tracking prevents stale async loads from attaching.

## The Inventory Doll

Inside the inventory modal, the same `useEquipment` composable powers a mini-canvas preview of the character with current weapons attached. It runs in `CharacterPreviewModel` against a derived equipment ref:

```ts
const equipment = computed(() => gameStore.derivedEquipment(props.characterId))
useEquipment(rig, equipment)
useCharacterAnimations(rig, rigSize) // plays idle automatically
```

See [Inventory UI Components](/inventory/ui-components) for the modal layout and slot-by-slot doll arrangement.

## Why GLTFLoader Instead of useGLTF

`useGLTF` resolves the path once via `toValue()` and doesn't re-load on reactive path changes. Equipment needs dynamic loading — weapons can be swapped at any time. `GLTFLoader.loadAsync()` gives the composable full control over when and what to load.

::warning
Weapon GLBs compressed with Draco require `DRACOLoader`. The composable configures this automatically using Google's CDN decoder.
::
