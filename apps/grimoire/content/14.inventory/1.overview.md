The inventory system models **items as first-class entities**. There is no separate "inventory list" attached to a character — every weapon, potion, and stack of coins is an `EntityState` of `type: 'item'` that lives in the same store as characters and interactables. Ownership is expressed by a single field, `containerId`, pointing to whichever entity currently holds it.

::alert{type="info"}
**Across the layers:** the pure carry-weight math (`carryCapacity`, `totalWeight`, `encumbrance`) lives in `@artificer-forge/engine/core`; the item state and the `moveItem` primitive live in the game store in `@artificer-forge/engine/runtime` (along with `useInventory` / `useLoot`); the HUD widgets ship from `@artificer-forge/engine/ui`.
::

## Item-Entity Model

```ts
interface EntityState {
  id: string                    // runtime instance id
  templateId: string            // 'shortsword', 'healing_potion'
  type: 'item'
  containerId?: string | null   // owning entity id, null = in world
  slot?: EquipmentSlotKey       // when equipped, names the slot
  quantity?: number             // for stackables
  weight?: number               // per unit (kg)
  // ...template metadata copied at spawn: damage, effect, value, properties
}
```

An item's state is the answer to four questions:

| Field | Question |
|-------|----------|
| `containerId` | Who owns it? (`null` = world) |
| `slot` | Is it equipped? Where? |
| `quantity` | How many in this stack? |
| `position` | Where in the world? (only when `containerId === null`) |

## Containers

Any entity with a `container` annotation can hold items. Containers come in three flavors:

| Container | Capacity | Where it comes from |
|-----------|----------|---------------------|
| Character bag | `50 + strength * 5` kg | All party characters |
| Chest | unlimited | `type: interactable, subtype: container` |
| Corpse | unlimited | Character whose `hp <= 0` or `dead === true` |

Capacity is computed on demand — characters don't store it. Chests and corpses use `capacity: undefined`, which means no weight check.

## States An Item Can Be In

```
in a bag        →  containerId: <character>, slot: undefined
equipped        →  containerId: <character>, slot: 'mainHand'
in a chest      →  containerId: <chest>,     slot: undefined
on a corpse     →  containerId: <corpse>,    slot: 'mainHand' (still equipped)
on the ground   →  containerId: null,        position: { x, y, z }
```

Equipment lives in the same container as the bag — the `slot` field is what distinguishes "in my pack" from "on my belt". This is why `itemsIn(characterId)` skips slotted items by default; the bag and the equipment row are two views of the same set.

## The moveItem Primitive

Every inventory operation funnels through `moveItem`. Equip, drop, pickup, transfer, swap — all of them call `moveItem` with different targets:

```ts
gameStore.equipItem(itemId, 'mainHand')
// → moveItem(itemId, { containerId: ownerId, slot: 'mainHand' })

gameStore.dropItem(itemId, position)
// → moveItem(itemId, { containerId: null, position })

gameStore.transferItem(itemId, otherCharacterId)
// → moveItem(itemId, { containerId: otherCharacterId })
```

This is the most important design choice in the system. Capacity checks, stack merging, slot validation, encumbrance sync, and partial-stack splitting all happen in **one place**. See [Store API](/inventory/store-api) for the full contract.

## Why Items-As-Entities

The old model attached a `mainHand: 'shortsword'` map to each character. It was simple but couldn't answer common questions:

- "How many potions does the party carry?" — requires a separate inventory array
- "Drop this sword on the ground" — items can't exist without an owner
- "Loot this corpse" — there's no shared item identity to transfer
- "Stack of arrows that splits when you give 5 away" — needs per-stack identity

Treating items as entities makes the world uniform: a sword on the floor, a sword in a chest, and a sword in your hand are the same record with a different `containerId`/`slot`. Scene save/load, loot transfers, and drag-and-drop all reduce to "change one field".

## Spawning Items

Items spawn one of three ways:

```ts
// 1. Standalone — e.g. loot, world drop
await gameStore.spawnItemEntity('healing_potion', {
  containerId: null,
  position: { x: 2, y: 0, z: 1 },
  quantity: 3,
})

// 2. As character starting equipment (via template.equipment)
//    Happens automatically inside spawnFromTemplate.
await gameStore.spawnFromTemplate('hero', spawn)

// 3. As chest/corpse loot (via template.lootTable)
//    Also automatic — rolled at spawn time.
await gameStore.spawnFromTemplate('chest_common', spawn)
```

`spawnItemEntity` reads the YAML template, copies item metadata onto the instance, and registers it in the entity store. From that point on it behaves like any other entity.
