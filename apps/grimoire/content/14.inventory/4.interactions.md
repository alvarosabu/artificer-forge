Inventory interactions are implemented as **shared composables** (`createSharedComposable` from VueUse). State is global — opening the modal anywhere, dragging an item from any cell, and right-clicking on any equipment slot all hit the same singletons.

## Composables at a Glance

| Composable | What it owns |
|------------|--------------|
| `useInventory` | Modal open state, focused character, `i` hotkey |
| `useItemDrag` | The currently-dragged item (singleton) |
| `useItemContextMenu` | Right-click menu position + actions for one item |
| `useLoot` | Loot popover position + container being looted |

All four are `createSharedComposable`, so every consumer reads the same reactive state.

## Opening the Modal — useInventory

```ts
const { isOpen, open, close, toggle, focus, focusedCharacter } = useInventory()

open()                 // focus party leader
open(characterId)      // focus a specific character
focus(otherCharacterId) // change focus without re-opening
```

### Hotkey

The composable registers a global `onKeyStroke('i')` listener that toggles the modal — unless the user is typing in an `<input>` or `<textarea>`. `Escape` closes it.

```ts
onKeyStroke('i', (e) => {
  const tag = (e.target as HTMLElement)?.tagName
  if (tag === 'INPUT' || tag === 'TEXTAREA') return
  e.preventDefault()
  toggle()
})
```

::alert{type="info"}
Because the composable is shared, the hotkey is registered exactly once even if multiple components import `useInventory`.
::

### Opening from the Party Panel

Click on a portrait to open the modal focused on that character:

```vue
<button @click="useInventory().open(member.id)">
  <img :src="member.portrait" />
</button>
```

## Drag-and-Drop — useItemDrag

A minimal singleton holding the item currently being dragged. Both ends of every drag go through it.

```ts
const itemDrag = useItemDrag()

// On dragstart of a cell:
itemDrag.start(item)

// On drop or dragend:
itemDrag.end()
```

The store mutation happens in the **drop target**, not the source. The source's only job is to publish "I'm dragging this".

### Drop Targets

| Target | Drop calls |
|--------|-----------|
| `EquipmentSlot` | `moveItem(id, { containerId, slot })` — slot-typed equip |
| `BagGrid` | `moveItem(id, { containerId })` — unequip or move to bag |
| `PortraitDropZone` | `transferItem(id, characterId)` — give to another party member |

Each target uses native HTML5 drag events plus VueUse helpers like `useDropZone` to add a hover ring.

### Cross-Character Transfer

`PortraitDropZone` wraps any DOM element (typically a party-panel portrait) as a drop target that transfers the item:

```vue
<InventoryPortraitDropZone :character-id="member.id">
  <PartyPortrait :member="member" />
</InventoryPortraitDropZone>
```

Drag a sword from the leader's bag onto another portrait — the store reparents the item to that character.

### Visual Feedback

Every drop zone has three visual states driven by `isValidTarget` + `isOverDropZone`:

- **idle** — dragging in progress, but cursor isn't here: dashed gold ring
- **valid hover** — pulse / scale-up + filled gold ring
- **invalid hover** — red ring (e.g. dragging an amulet onto `mainHand`)

Validity is checked via `isItemTypeForSlot` for equipment slots and "different container" for bags. The store still re-validates on drop — the UI hint is advisory.

## Context Menu — useItemContextMenu

Right-clicking an item opens a context menu whose options depend on **where the item lives**.

```ts
const itemMenu = useItemContextMenu()

function onContext(event: MouseEvent, item: EntityState) {
  itemMenu.openAt(event, item.id)
}
```

Render the menu once at the root:

```vue
<InventoryItemContextMenu />
```

### Actions by Owner

`actionsFor(item)` inspects the item's owner and returns grouped `DropdownMenuItem`s:

| Owner | Actions offered |
|-------|-----------------|
| Lootable (chest / dead character) | **Pick Up** → recipient is selected entity or leader |
| Bagged (alive character, not equipped) | **Equip (slot)** for each valid slot on the owner |
| Equipped (alive character) | **Unequip** |
| Any (with other party members) | **Transfer to…** submenu listing party members |
| Bagged (alive character, not equipped) | **Drop** at the owner's position |

Actions are split into primary and secondary groups; the menu renders them with a separator.

::alert{type="warning"}
A dead character's items are *displayed as equipped* (the doll still wears them), but the menu only offers **Pick Up** — you cannot unequip an item on a corpse. Looters take it directly.
::

## Loot Popover — useLoot

A small composable for the loot popover position. Driven by the world's context menu (right-click a chest or corpse).

```ts
const loot = useLoot()
loot.open(containerId, screenX, screenY)
loot.close()
```

See [Loot](/inventory/loot) for the full chest/corpse pickup flow.

## Putting Interactions Together

Inside a scene page, the wiring is roughly:

```ts
const { onAction } = useContextMenu()
const loot = useLoot()

onAction((action, entityId) => {
  switch (action) {
    case 'loot':   walkToAndLoot(entityId); break
    case 'pickup': gameStore.pickupItem(entityId, leader); break
  }
})
```

Everything else — the modal, the menu, the drag — is already global. The page just needs to render the singletons and handle world-side actions.
