Two cross-cutting concerns sit on top of the item-entity model: **stacking** (multiple units in one entity) and **weight** (a soft cap that triggers a status effect).

## Stackables

A stackable item carries a `quantity` and a `maxStack` on its instance, copied from the template.

```yaml
# content/entities/items/consumables/healing_potion.yaml
templateId: healing_potion
type: item
subtype: consumable
name: Healing Potion
weight: 0.5
stackable: true
maxStack: 10
```

A character holding 7 potions has **one** `EntityState` with `quantity: 7`. The bag grid renders a small badge on the cell when quantity > 1.

### Merge on Move

When `moveItem` lands a stackable item into a container, it first looks for an existing partial stack of the same `templateId`:

```ts
if (target.containerId && !target.slot && item.stackable) {
  const existing = itemsIn(target.containerId).find(e =>
    e.templateId === item.templateId
    && (e.quantity ?? 1) < (e.maxStack ?? 1),
  )
  if (existing) {
    const room = (existing.maxStack ?? 1) - (existing.quantity ?? 1)
    const merged = Math.min(room, moveQty)
    updateEntity(existing.id, { quantity: existing.quantity + merged })
    // If the whole moved stack fit, remove the source entity.
    // If not, leave the leftover units on the source container.
  }
}
```

Two consequences worth knowing:

1. **Stacks never split across multiple entities in the same bag.** If you have a stack of 7 and pick up 5 more (maxStack 10), you end up with one stack of 10 and one of 2 — not two stacks of 6 next to each other.
2. **Overflow stays put.** If the target stack has no room, the source isn't moved at all. The caller gets `{ ok: true }` only if at least one unit moved.

### Partial Transfer

Pass `quantity` to `moveItem` to split a stack:

```ts
gameStore.moveItem(potionStackId, {
  containerId: otherCharacterId,
  quantity: 3,
})
```

This spawns a new entity for the 3 units, decrements the source, and (if the target has matching stacks) tries to merge them in.

::alert{type="info"}
The current playground UI does not expose split-quantity to the player — full stacks are transferred. The primitive supports it, so a "split…" dialog can be added without store changes.
::

## Weight & Capacity

Weight is per-unit, declared on the item template (`weight: 0.5` for a potion). The store derives totals on demand:

```ts
weightOf(characterId)     // Σ weight × quantity across all items (incl. equipped)
capacityOf(characterId)   // 50 + strength × 5 kg
encumbranceOf(characterId) // ratio, may exceed 1.0
```

### Capacity Formula

Capacity is computed from the character's `stats.strength`:

```
capacity = 50 + strength × 5  (kg)
```

A baseline character with Strength 10 carries 100 kg before becoming encumbered. There is no per-character override field — strength is the only knob.

### Chests and Corpses

Containers can set `container.capacity` to a number or leave it `undefined`. Currently chests and corpses are **unlimited** (`undefined`), so loot piles never reject incoming items. Only characters enforce the cap.

## Encumbrance Status Effect

When `encumbranceOf(characterId)` crosses 1.0, the store adds an `encumbered` status effect; when it drops back below, the effect is removed.

```ts
function syncEncumbrance(characterId) {
  const ratio = encumbranceOf(characterId)
  const hasEffect = character.statusEffects?.some(e => e.id === 'encumbered')
  if (ratio >= 1 && !hasEffect)        addStatusEffect(characterId, 'encumbered')
  else if (ratio < 1 && hasEffect)     removeStatusEffect(characterId, 'encumbered')
}
```

`syncEncumbrance` runs **after every successful `moveItem`** on both the source and target characters. You normally don't call it directly, but it's exported in case a stat change (Strength buff/debuff, level-up) needs to re-check.

### Visual Signal

The `WeightBar` component colors itself by ratio:

| Ratio | Color | Effect |
|-------|-------|--------|
| < 0.8 | gold | normal |
| 0.8 – 1.0 | warning yellow | nearing limit |
| ≥ 1.0 | error red | `encumbered` status applied |

See [Status Effects](/characters/status-effects) for how the visual badge appears on the character.

## Hard vs Soft Capacity

The cap is **soft on character→character transfers** (the bag refuses) but **never blocks loot from spawning** — chests can hold unlimited weight, and starting equipment ignores the cap entirely (it's set before any weight is computed). This keeps designers free to place rich loot without micromanaging carry weight at spawn time. Players still have to deal with it during pickup.

## Common Recipes

### Give the party 10 potions

```ts
await gameStore.spawnItemEntity('healing_potion', {
  containerId: gameStore.party.leader!,
  quantity: 10,
})
```

If the leader already has potions, they'll merge into existing stacks up to `maxStack`.

### Force-clear encumbrance after a Strength buff

```ts
gameStore.syncEncumbrance(characterId)
```

### Check whether a pickup will fit

```ts
const ok = gameStore.weightOf(charId) + item.weight * qty <= gameStore.capacityOf(charId)
```

Or just call `moveItem` and read the result — the store does the same math.
