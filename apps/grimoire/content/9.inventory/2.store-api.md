All inventory mutations go through `useGameStore`. The store exposes one primitive (`moveItem`) and a handful of wrappers and queries.

## The MoveResult Contract

Every mutation returns a `MoveResult`. Callers check `result.ok` and surface the failure reason — the store never throws on a normal user action like "bag is full".

```ts
type MoveFailReason =
  | 'no-slot'         // target has no such equipment slot
  | 'wrong-type'      // item subtype doesn't match the slot
  | 'over-capacity'   // would exceed container weight capacity
  | 'occupied'        // (reserved)
  | 'not-found'       // item or container missing
  | 'invalid-target'  // missing position, etc.

type MoveResult =
  | { ok: true }
  | { ok: false, reason: MoveFailReason }
```

## moveItem

The one primitive that everything else builds on.

```ts
moveItem(itemId: string, target: {
  containerId: string | null
  slot?: EquipmentSlotKey
  position?: Position
  quantity?: number
}): MoveResult
```

What it does, in order:

1. **Resolve target** — fetch the container entity (or require `position` if `containerId === null`).
2. **Validate slot** — check the slot exists on the container, the item subtype matches (`isItemTypeForSlot`), and **unequip any current occupant** of that slot (equip-swap).
3. **Capacity check** — for non-slot moves into a container with `container.capacity`, reject if `weightOf(target) + item.weight * qty > capacity`.
4. **Merge into existing stack** — if `item.stackable` and a partial stack of the same `templateId` exists in the target, fill it. Overflow stays on the source.
5. **Partial split** — if `quantity < item.quantity`, spawn a new entity for the moved portion and decrement the source.
6. **Whole reparent** — otherwise just update `containerId`, `slot`, and (if dropping) `position`.
7. **Sync encumbrance** — recompute the `encumbered` status effect on both source and target characters.

## Wrappers

These exist for readability — they all forward to `moveItem`.

```ts
gameStore.equipItem(itemId, slot)
// → moveItem(itemId, { containerId: ownerId, slot })

gameStore.unequipItem(itemId)
// → moveItem(itemId, { containerId: ownerId, slot: undefined })

gameStore.dropItem(itemId, position)
// → moveItem(itemId, { containerId: null, position })

gameStore.pickupItem(itemId, characterId)
// → moveItem(itemId, { containerId: characterId })

gameStore.transferItem(itemId, toCharacterId)
// → moveItem(itemId, { containerId: toCharacterId })
```

## Queries

Read-only helpers that derive answers from the entity map. They scan all entities, so don't call them in hot loops — wrap in `computed`.

| Query | Returns |
|-------|---------|
| `itemsIn(containerId, { includeEquipped? })` | Items where `containerId` matches. Skips slotted items unless asked. |
| `equippedAt(characterId, slot)` | The single item in that slot, or `null`. |
| `weightOf(characterId)` | Sum of `weight * quantity` for all items in the container (including equipped). |
| `capacityOf(characterId)` | `50 + strength * 5` kg. |
| `encumbranceOf(characterId)` | `weight / capacity` ratio (0–∞). |
| `derivedEquipment(entityId)` | `{ mainHand, offHand }` of equipped template ids. Used by the doll. |
| `isItemTypeForSlot(item, slot)` | Whether `item.subtype` is allowed in `slot`. |

### Example: render a bag

```ts
const items = computed(() => gameStore.itemsIn(props.characterId))
const weight = computed(() => gameStore.weightOf(props.characterId))
const capacity = computed(() => gameStore.capacityOf(props.characterId))
```

### Example: render an equipment slot

```ts
const item = computed(() => gameStore.equippedAt(characterId, 'mainHand'))
```

## Slot Type Matrix

`isItemTypeForSlot` enforces the mapping between `slot` and `item.subtype`:

| Slot | Accepted subtype |
|------|------------------|
| `mainHand`, `offHand` | `weapon` |
| `helmet` | `helmet` |
| `armor` | `armor` (contributes `armor.physical` / `armor.magical` &mdash; see [Combat &rarr; Armor](/combat/armor)) |
| `gauntlets` | `gauntlets` |
| `boots` | `boots` |
| `amulet` | `amulet` |
| `ring1`, `ring2` | `ring` |

::alert{type="info"}
Subtype enforcement happens in `moveItem`, not at the UI layer. Drag-and-drop and the context menu are free to attempt invalid moves — they'll just get back `{ ok: false, reason: 'wrong-type' }`.
::

## spawnItemEntity

Standalone spawn — used by loot tables, starting equipment, and any code that drops an item into the world.

```ts
spawnItemEntity(templateId: string, opts: {
  containerId?: string | null
  slot?: EquipmentSlotKey
  position?: Position
  quantity?: number
}): Promise<string | null>
```

Returns the new entity id, or `null` if the template is missing or not of `type: 'item'`. It reads the YAML template via `queryCollection('entities')` and copies item metadata (`damage`, `armor`, `effect`, `weight`, `value`, `stackable`, `maxStack`, etc.) onto the instance.

::alert{type="info"}
Items can carry an `armor: { physical?, magical? }` block that contributes to the bearer's armor pools while equipped. See [Combat &rarr; Armor](/combat/armor) for the full derivation and damage flow.
::

## Equip-Swap

Calling `equipItem` on a slot that's already filled does **not** fail. The current occupant is silently unequipped (moved to the same bag with `slot: undefined`) and the new item takes its place. This matches the behavior players expect when dragging a new weapon onto an occupied slot.

```ts
// Slot already has a shortsword. Equipping the axe swaps them.
gameStore.equipItem(axeId, 'mainHand')
// → axe is now in mainHand, shortsword is now in the bag
```

## syncEncumbrance

Called automatically after every successful `moveItem`. Compares `encumbranceOf(characterId)` to `1.0` and adds or removes the `encumbered` status effect. You normally don't call this directly — but it's exported in case a stat change (e.g. losing Strength) needs to re-check.

See [Stackables &amp; Weight](/inventory/stackables-and-weight) for the full encumbrance flow.
