title: Dialog Engine
description: useDialogEngine and the dialog store — the real API for opening, advancing, and resolving a dialog tree.

The dialog runtime splits into two layers, both exported from `@artificer-forge/engine/runtime`:

- **`useDialogStore`** — the stateful Pinia store. Holds the active tree, the current node, and the player-facing getters. This is what your page and the HUD talk to.
- **`useDialogEngine`** — pure, stateless logic the store leans on: building the dialog context, resolving text, gating choices, rolling skill checks, applying effects. You rarely call it directly; the store wires it up for you.

```ts
import { useDialogEngine, useDialogStore } from '@artificer-forge/engine/runtime'
```

## The store

### State

| Member | Type | Meaning |
|--------|------|---------|
| `activeTree` | `DialogTree \| null` | The currently playing tree (`dialogId`, `startNode`, `nodes`) |
| `currentNodeId` | `string \| null` | Id of the node being shown |
| `speakerEntityId` | `string \| null` | The resolved speaker entity for this dialog |
| `history` | `DialogHistoryEntry[]` | Lines + chosen replies, in order |
| `lastCheck` | `CheckRoll \| null` | Most recent skill-check result (auto-clears after ~2.5s for the panel flash) |

### Getters

| Getter | Type | Meaning |
|--------|------|---------|
| `isOpen` | `boolean` | True while a tree + node are active |
| `currentNode` | `DialogNode \| null` | The active node object |
| `currentText` | `string` | Resolved text for the current node (honors `textVariants`) |
| `currentChoices` | `ResolvedChoice[]` | Visible, condition-evaluated choices (hidden ones filtered out) |
| `currentSpeakerEntity` | `EntityState \| null` | The speaker entity, resolved by id then `templateId` |

### Actions

| Action | Signature | Effect |
|--------|-----------|--------|
| `open` | `(dialogId: string, speakerId: string \| null) => Promise<void>` | Resolves the tree via the game store, blocks input, enters the start node |
| `choose` | `(choiceIndex: number) => void` | Picks a choice: applies its effects, rolls any check, branches to the next node |
| `close` | `() => void` | Tears down the dialog and unblocks input |

::note
`open` is re-entry safe — if a dialog is already open it returns early, so a repeated interaction can't clobber live state. The tree is fetched through `gameStore.resolveDialog(dialogId)`; a missing tree logs a warning and bails.
::

## Starting and advancing a dialog

```ts
const dialogStore = useDialogStore()

// Start: pass the dialog id and the speaking entity (e.g. an NPC instance id).
await dialogStore.open('elder_greeting', npcEntityId)

// The HUD reads these reactively:
dialogStore.currentText      // line for the current node
dialogStore.currentChoices   // ResolvedChoice[] the player can pick

// Advance by choosing — index is the choice's `index` field, not the array position.
dialogStore.choose(choice.index)

// Bail out at any point.
dialogStore.close()
```

`choose` walks one step:

1. Records the player's reply on the latest `history` entry.
2. Applies the choice's `effects` (costs/triggers fire for every choice, before branching).
3. If the choice has a `check`, rolls it (`rollCheck`), flashes the result, then follows `onSuccess` / `onFailure` — applying that branch's effects and entering its `next`.
4. Otherwise follows `choice.next`. A missing `next` (or the sentinel `__end`) ends the dialog.

## Node and choice shape

The engine's types (from `useDialogEngine`) describe an authored node:

```ts
interface DialogNode {
  speaker?: string                     // templateId of the speaker
  text: string
  textVariants?: { if: Record<string, unknown>, text: string }[]
  cameraShot?: 'three-quarter' | 'over-shoulder' | 'closeup' | 'wide' | 'two-shot'
  cameraTarget?: string
  choices?: DialogChoice[]
  effects?: Record<string, unknown>[]  // on-entry effects
}

interface DialogChoice {
  text: string
  tagPrefix?: string                   // e.g. [PERSUASION] label
  conditions?: Record<string, unknown>[]
  check?: { skill: string, dc: number, advantage?: boolean }
  next?: string
  onSuccess?: { next?: string, effects?: Record<string, unknown>[] }
  onFailure?: { next?: string, effects?: Record<string, unknown>[] }
  effects?: Record<string, unknown>[]
  lockedDisplay?: 'hide' | 'lock'      // hide a failed-condition choice, or show it locked
}
```

## How the engine resolves a choice

`useDialogEngine` is where the logic lives. The store calls these for you, but they describe exactly what happens:

| Function | Purpose |
|----------|---------|
| `buildContext(speakerId)` | Builds `DialogContext` — `player` (party leader), `speaker`, `partyEntities`, `gameStore` |
| `resolveText(node, ctx)` | Returns the first matching `textVariants` entry, else `node.text` |
| `availableChoices(node, ctx)` | Evaluates each choice's `conditions` into a `ResolvedChoice` (`available`, `lockedReason`, `hide`) |
| `rollCheck(check, ctx)` | d20 (+ advantage) plus the skill's stat modifier vs `dc` → `CheckRoll` |
| `applyEffects(effects, ctx)` | Mutates game state; returns `{ endDialog }` |
| `evaluateConditions(conditions, ctx)` | True if every condition passes |

### Conditions

Choices gate on the `DialogContext`. Supported keys: `flag`, `notFlag`, `class`, `race`, `faction`, `hasAbility`, `hasStatusEffect`, `stat` (with `gte`/`lte`/`gt`/`lt`/`eq` comparisons), `partyHas`, `hasItem` (`{ id, qty }`). A failing condition produces a human-readable `lockedReason` (e.g. `strength ≥ 14`, `Need 10× gold`).

### Skill checks

`rollCheck` maps a skill to a D&D 5e ability (e.g. `persuasion → charisma`, `athletics → strength`), reads the player's stat, derives the modifier `floor((stat - 10) / 2)`, rolls a d20 (twice and takes the max when `advantage`), and compares `roll + modifier` to `dc`. The result lands in `lastCheck` and the success/failure branch decides the next node.

### Effects

`applyEffects` runs an array of effect objects against the context. Supported keys: `setFlag`, `clearFlag`, `addStatusEffect`, `removeStatusEffect`, `learnAbility`, `giveItem` (`{ id, qty }`), `takeItem` (`{ id, qty }`), and `endDialog`. Effects fire on node entry and on choosing; `endDialog` closes the conversation immediately.
