The **studio** is the off-screen `TresCanvas` that actually renders a portrait. It is mounted once (e.g. inside the HUD) and stays passive until `usePortraitStudio()` hands it a subject to render. Its canvas lives at `left: -9999px` with `opacity: 0` — off-screen rather than `display: none` so the render loop keeps running and `preserveDrawingBuffer` keeps the frame readable for the capture.

```ts
import {
  PortraitStudio,
  PortraitSubject,
  PortraitBackground,
  PortraitLights,
} from '@artificer-forge/engine/runtime'
```

## `PortraitStudio`

The composition root. Mount it once. It owns the camera, the lights and the backdrop, and conditionally mounts a `PortraitSubject` whenever the shared studio state has an active descriptor:

```vue
<script setup lang="ts">
import { PortraitStudio } from '@artificer-forge/engine/runtime'
</script>

<template>
  <PortraitStudio />
</template>
```

Internally it:

- Reads `active`, `captured` and `failed` from `usePortraitStudio()` (the singleton producer/consumer pair).
- Holds a `framing` (camera position, look-at, fov). It starts on a neutral placeholder and is overwritten by the subject's `@head` event the instant the model's bounds resolve — so placeholder values never bake.
- Drives the camera **imperatively** via `cam.lookAt(...)` in a `watchEffect`, because Tres does not re-apply `:look-at` when only the position changes (a yaw-orbited frame would otherwise capture facing front).
- Wraps the subject in `<Suspense>` and uses `onErrorCaptured` to route a load failure (bad model URL, Draco error) into `failed()`, so a broken bake settles the promise instead of deadlocking the queue.

## `PortraitSubject`

Loads the GLTF, plays the idle animation, frames the head, and captures the frame. It is a child of the canvas (inside the `<Suspense>` boundary) because that is where the Tres context resolves.

| Prop | Type | Default | Purpose |
|------|------|---------|---------|
| `descriptor` | `PortraitSubjectDescriptor` | — | What to render: `model`, `rig`, `equipment`, optional `background` |
| `autoCapture` | `boolean` | `true` | When `false`, render only (no `toDataURL` bake) — used by the portrait lab for a live preview |

Emits:

| Event | Payload | Purpose |
|-------|---------|---------|
| `head` | `[Vec3, number]` | Head-mesh world-box centre + height — drives the portrait camera |
| `bounds` | `[Vec3, Vec3]` | Full-model bounds — used by the lab for the orbit target / readout only |
| `captured` | `[string]` | The PNG data URL once the frame is read |
| `failed` | `[unknown]` | A load or capture error |

The descriptor shape, exactly as defined in `usePortraitStudio.ts`:

```ts
interface PortraitSubjectDescriptor {
  model: string
  rig: string
  equipment: Equipment
  background?: string // per-character backdrop URL; undefined = transparent
}
```

::alert{type="info"}
The subject gates on the `IDLE_A` action specifically, not on "any action loaded". Animation packs load **outside** `Suspense` and fill `actions` incrementally — signalling on the first available action risks capturing a bind/T-pose. After idle settles (~200ms) the world matrices are valid and the head box is accurate.
::

## Head-mesh framing

Portraits are framed off the **head mesh's** world box, not the whole-model bounding box and not the head *bone*. The head mesh is consistent across bodies, equipment and poses, and it scales to the actual head size — keying off the head bone overshoots tall rigs (a werewolf's head bone sits high on long legs, but its head isn't proportionally bigger).

`frameFromHead(center, headHeight, opts?)` turns a head box into a `PortraitFraming`. Offsets are expressed as fractions of head height, so the same knobs hold across rig sizes. The defaults live in `PORTRAIT_CAMERA`:

```ts
export const PORTRAIT_CAMERA = {
  headLift: 0,            // shift look point above/below head centre
  viewHeight: 1.3,        // vertical slice to fit, as a multiple of head height
  fov: 30,                // field of view (deg)
  cameraHeightOffset: 0.05, // camera height vs look point
  yaw: 20,                // horizontal orbit angle for a 3/4 view
}
```

Pass a `FrameOptions` partial as the third argument to override any of these (the portrait lab uses this to retune framing live). The model is assumed to face +Z (Three/GLB convention), so the camera is placed in front of the face.

## `PortraitLights`

Shared three-point dramatic lighting, used by both the bake studio and the portrait lab so the preview matches the baked output. The drama comes from a high key-to-fill ratio plus warm/cool contrast:

| Light | Role | Intensity | Colour |
|-------|------|-----------|--------|
| `TresAmbientLight` | base lift | `0.18` | — |
| `TresDirectionalLight` | warm key, high-right | `2.5` | `#fff1dc` |
| `TresDirectionalLight` | cool fill, opposite | `0.35` | `#9fb8ff` |
| `TresDirectionalLight` | cool rim, backlight | `2.6` | `#cfe0ff` |

## `PortraitBackground`

A flat backdrop rendered as a **child of the portrait camera**, so it is camera-local: a plane at `-Z` always faces the lens and fills the frame regardless of how the camera was orbited.

| Prop | Type | Purpose |
|------|------|---------|
| `framing` | `PortraitFraming` | Current camera framing — used to size + place the plane in the frustum |
| `src` | `string` | Backdrop texture URL |

It uses an unlit `TresMeshBasicMaterial` with `:tone-mapped="false"`, so the three-point portrait lighting never tints the backdrop, and the texture is decoded in `SRGBColorSpace`. The plane is pushed well behind the subject (so the body never clips through) and sized to the vertical FOV at that depth.

### Resolving the backdrop

Each character can author a backdrop on its entity YAML via `portraitBackground`. `resolvePortraitBackground()` resolves it, falling back to a shared default so every portrait gets a backdrop:

```ts
import { resolvePortraitBackground, DEFAULT_PORTRAIT_BACKGROUND } from '@artificer-forge/engine/runtime'

resolvePortraitBackground('/img/portraits/bgs/cave.png') // -> the authored URL
resolvePortraitBackground(undefined) // -> DEFAULT_PORTRAIT_BACKGROUND
```

::alert{type="warning"}
The backdrop is part of the **appearance**, so it is folded into the [signature](/portraits/signature). Changing or adding a backdrop re-bakes the portrait.
::
